## Origin — Why we pivoted

The sales team's query tool (current branch: suggest → structured params → compiled SQL) does not feel like working with Claude directly. The user framed it sharply: *"when I work with you in the context of the repo, you can generate reliable reports and ask great clarifying questions when you're not sure what I'm looking for — why doesn't the query tool feel like that?"*

Grounded in `src/app/api/ai/query/suggest/route.ts`, the architectural gap is:

1. **One-shot, not a conversation.** `tool_choice: { type: "tool", name: "run_query" }` forces Claude to emit structured params on the first turn. It cannot ask a clarifying question — the API won't let it respond with text.
2. **No tool loop.** Claude has zero ability to peek at the data (distinct values, row counts, samples) before guessing a query. In Claude Code, this exploration is how I *earn* reliable answers.
3. **No session memory.** `conversationId` is accepted and logged but never fed back in. "Like the last one but for Texas" has nothing to reference.
4. **Output shape is a straitjacket.** The only tool is `run_query` with a fixed params schema. Even when the request is ambiguous, Claude must commit.
5. **DSL is narrower than the question space.** The `params-to-sql.ts` compiler can't express OR branches, `CASE`, `EXISTS`, subqueries, CTEs, window functions, math across aggregates, `HAVING`, self-joins, or `UNION`. A huge fraction of rep questions — YoY growth %, win rates, set-difference queries, per-student ratios — silently fall outside the grammar.

The redesign replaces all five with one architectural decision: **let Claude run as a multi-turn agent with real tools and emit full SQL against a readonly pg role.**

## The pivot — chips describe the query

**Previous decision (now reversed):** chips *are* the query. Claude emits structured params. Compiler produces SQL. The builder is WYSIWYG: what you see is what runs.

**Current decision:** chips *describe* the query. Claude emits full SQL and a human-readable summary. The UI renders the summary as chips. Users never see SQL.

**Tradeoff accepted:** the chip representation is lossy (Claude interprets intent into rep-friendly language) rather than exhaustive. The old "chips = query" trust surface is replaced by "a human summary of what ran + the results + SQL never visible."

**Why it's the right trade:**

- Expressive range unlocks. Real rep questions become answerable.
- Safety rail shifts from "narrow grammar" to "readonly pg role + statement_timeout." The `query_tool_readonly` role is already provisioned (db-readiness Tasks 3–4, committed).
- "Never show SQL to reps" — unchanged. That rule is the user's, and it is load-bearing; reps can't read or approve SQL and asking them to would kill adoption.
- Saved reports still zero-token on rerun. They now store SQL + summary instead of structured params. Same economics.
- The tool starts feeling like Claude Code.

## Design decisions

| # | Question | Decision | Rationale |
|---|---|---|---|
| D1 | Query grammar | Full SQL generated by Claude against `query_tool_readonly` | Expressive range, matches how Claude reasons about data elsewhere. DSL was strictly a subset. |
| D2 | Agent loop | Multi-turn, no forced `tool_choice`, Claude can emit text or tool calls | One-shot forced tool use kills clarifying questions. Matches Claude Code. |
| D3 | Tool surface | `list_tables`, `describe_table`, `search_metadata`, `get_column_values`, `count_rows`, `sample_rows`, `run_sql`, `search_saved_reports`, `get_saved_report` | Derived from "what I actually do when querying the repo in Claude Code" — metadata grep, distinct-value peek, row count sanity checks, sample inspection before committing. |
| D4 | Stop condition | `run_sql` is terminal per turn. Further refinement happens as a new user turn. | Prevents spiral-refinement past what the user asked. Matches how it feels to work with Claude: you ask, I converge, I deliver, you react. |
| D5 | SQL error handling | Claude sees the pg error as a tool result, retries up to 2x, then surrenders with a human-language message ("I tried to run that but the query failed; can you tell me more about…"). | Reps can't parse `column "foo" does not exist`. Matches Claude Code debugging loop. |
| D6 | Chip summary generation | Option A — Claude returns `run_sql` with a `summary` sibling field (structured: source, filters, columns, sort, limit). | Reads as rep language, not EXPLAIN output. An AST summarizer would be faithful but unusable for non-trivial queries. |
| D7 | Chip summary validation | Lightweight server-side check: every summary filter value must literally appear as a parameter binding in the SQL; summary column count must equal the SELECT's column count. | Catches the worst hallucinations (chip says "Texas" but SQL filters Ohio) without trying to parse arbitrary SQL. |
| D8 | Chip editing | Option C — each chip edit round-trips through the agent loop. Edit actions (remove_filter / change_filter / add_column / etc.) become structured user turns; Claude regenerates SQL + summary. | Infinite flexibility, no AST manipulation, consistent with chips-describe. Token cost per edit is small (short turn). |
| D9 | Saved reports storage | Store SQL string + summary + name + ownership. Reruns execute SQL directly against readonly pool — zero Claude calls. | Same token economics as the old structured-params plan, strictly larger expressive range. |
| D10 | "Never show SQL" | Unchanged. No preview, modal, toast, or expandable section. | Rep audience. |
| D11 | "No IDs in output" enforcement | Two layers: (a) system prompt forbids SELECTing ID columns unless user asked by name; (b) UI hides columns whose metadata has `format: "id"` behind a "show technical columns" toggle. | Prompt is ~90% reliable; UI layer is cheap defense-in-depth. |
| D12 | Editing a saved report | Opening and editing chips on a saved report prompts "update saved report / save as new"; no silent mutation. | Saved reports are stable artifacts reps share; mutation surprises are bad. |
| D13 | Legacy compiler | `params-to-sql.ts`, `params-validator.ts`, `types.ts` (`QueryParams`), and the forced-tool-choice `suggest` route are deleted when Plan 2 lands. | No second code path; chips-are-the-query is dead. |

## Tool surface — full list

All tools are exposed to Claude inside the agent loop. All are read-only; `run_sql` enforces readonly at the pg role level.

### `list_tables()`
Returns the set of queryable tables from `TABLE_REGISTRY`, each with its description and tag set. Used for orientation at the start of a turn.

### `describe_table(name: string)`
Returns a table's columns (name, type, description), its `relationships` (join possibilities), warnings, and any `SEMANTIC_CONTEXT` entries scoped to it. This is Claude's primary schema-understanding tool.

### `search_metadata(query: string)`
**The most important tool.** Full-text search across column descriptions + `SEMANTIC_CONTEXT` Q&A entries, returning ranked matches with full prose. Mirrors what I do in Claude Code when I grep `district-column-metadata.ts` for a concept. Example: `search_metadata("bookings")` returns every column that mentions bookings with its description, plus any SEMANTIC_CONTEXT entries explaining closed-won detection, chain-dedup, min_purchase_bookings vs bookings, etc.

### `get_column_values(table: string, column: string, limit: int = 30)`
Returns distinct values for a column up to `limit`. Equivalent to `SELECT DISTINCT ... LIMIT` I run all the time to see what stage strings actually exist.

### `count_rows(where_sql: string, from_sql: string)`
Returns row count for a given FROM + WHERE fragment. Used to sanity-check a filter before building on it. Must be parameterized (not user input).

### `sample_rows(sql: string, limit: int = 5)`
Executes a provided SQL statement with an enforced LIMIT ≤ 20 and returns the rows. Used to peek at query shape before committing. Results go to Claude, not the UI.

### `run_sql(sql: string, summary: SummaryObject)` — terminal
Executes the final SQL against the readonly pool, returns rows to the UI, logs to `query_log`, renders `summary` as chips. Ends the turn. Only one `run_sql` per turn.

### `search_saved_reports(query: string)`
Full-text search of the user's saved reports by name and summary content. For "like the one I ran last week about Texas."

### `get_saved_report(id: string)`
Returns a saved report's SQL + summary + metadata. Claude can adapt it for a new variant.

## Agent loop behavior

```
user turn (chat or chip edit)
  → POST /api/ai/query { conversationId, message | edit_action }
  → load conversation history from prior turns
  → system prompt: tool definitions + abbreviated metadata + "never SELECT IDs" rule + "ask clarifying questions when ambiguous" guidance
  → Claude loop:
      while true:
        response = claude.messages.create(..., tools=[...])  // NO tool_choice
        if response has text → stream to chat, continue loop if tool_use also present
        if response has tool_use:
          if tool is run_sql:
            validate summary against SQL (D7)
            execute via readonly pool (retry budget exhausted on pg error → D5)
            log to query_log
            return rows + summary to UI
            BREAK (terminal)
          else if tool is exploratory (list_tables, describe_table, search_metadata, get_column_values, count_rows, sample_rows, search_saved_reports, get_saved_report):
            execute, append tool_result, continue loop
          if run_sql errored AND retries_remaining > 0:
            append pg error as tool_result, continue loop, decrement retries
  → response stream completes
  → UI renders: chat messages + chips + results table
```

### Loop bounds

- **Max exploratory tool calls per turn:** 20. Hard cap to prevent runaway. Emit human-language "I'm having trouble narrowing this down — can you tell me more about…" if exceeded.
- **Max SQL retries after `run_sql` error:** 2. Third failure surrenders to the user.
- **Statement timeout on readonly pool:** 30s (enforced at pg role level).
- **Max rows returned to UI:** 500 (enforced in `run_sql` — Claude must include `LIMIT ≤ 500`, server double-checks).

### Conversation threading

`conversationId` on the request is the key. Prior turns (user messages, Claude messages, tool calls and their results) are loaded and prepended. This is what makes "like that but for Texas" work.

Each turn writes to `query_log` with the `conversationId`, the user message, the final SQL (if any), the summary, execution time, and retry count.

## Chip summary model

Claude's `run_sql` tool call has two arguments:

```typescript
{
  sql: string;            // the SQL to execute against readonly pool
  summary: {
    source: string;       // rep-language description of what's being queried
    filters: ChipFilter[];
    columns: string[];    // rep-language column labels (from metadata where possible)
    sort: { column: string; direction: "asc" | "desc" } | null;
    limit: number;
  };
}

type ChipFilter = {
  label: string;   // e.g. "State"
  value: string;   // e.g. "Texas"
  // optional: operator for non-equality filters
  // e.g. label: "Bookings", operator: ">", value: "$50,000"
  operator?: string;
};
```

### Example

User asks: *"Texas districts we won this year but haven't renewed yet."*

```typescript
{
  sql: "SELECT d.name, doa.bookings FROM districts d JOIN district_opportunity_actuals doa ON d.leaid = doa.leaid WHERE d.state = $1 AND doa.school_yr = $2 AND doa.bookings > 0 AND NOT EXISTS (SELECT 1 FROM district_opportunity_actuals doa2 WHERE doa2.leaid = d.leaid AND doa2.school_yr = $3) ORDER BY doa.bookings DESC LIMIT 500",
  // bound params: ["TX", "2025-26", "2026-27"]
  summary: {
    source: "Districts with closed-won contracts",
    filters: [
      { label: "State", value: "Texas" },
      { label: "Won in", value: "FY26 (2025–26)" },
      { label: "Not yet renewed for", value: "FY27 (2026–27)" },
    ],
    columns: ["District name", "Bookings"],
    sort: { column: "Bookings", direction: "desc" },
    limit: 500,
  },
}
```

### Validation (D7)

Server enforces:
- Every `filter.value` that appears non-opaquely (i.e. "Texas", not "FY26 (2025–26)") must match a string or number in the SQL's bound parameters or explicit literals. For labels like "FY26 (2025–26)" that contain `2025-26`, the substring match suffices.
- `summary.columns.length === parsed(sql).select.columnCount`
- `summary.sort === null` OR the sorted column appears in the SELECT list.

Validation failure is surfaced to Claude as a tool_result error and counts against the retry budget (D5).

## Chip edit protocol (D8)

Each chip carries a stable `id`. Frontend interactions emit structured edit actions as user turns into the agent loop. The chat rail renders each as a natural-language user message for readability.

```typescript
type ChipEditAction =
  | { type: "remove_filter"; chipId: string; label: string }
  | { type: "change_filter"; chipId: string; label: string; from: string; to: string }
  | { type: "remove_column"; columnLabel: string }
  | { type: "add_column"; columnLabel: string }
  | { type: "change_sort"; column: string; direction: "asc" | "desc" }
  | { type: "remove_sort" }
  | { type: "change_limit"; from: number; to: number };
```

Rendering in chat rail:
- `remove_filter` → "Remove the Texas state filter"
- `change_filter` → "Change the school year filter from FY26 to FY25"
- etc.

Claude sees both the structured action (in context) and the rendered message. Regenerates SQL + summary. UI updates chips + reruns.

## Saved reports model

Schema (already migrated — `saved_reports` table exists per `prisma/migrations/...`):

Reports store:
- `id`, `userId`, `name`, `description`
- `sql` (the exact string to execute; no Claude call on rerun)
- `summary` (JSON — the chip model above; for display)
- `conversationId` (origin thread — for "open in chat to modify")
- `createdAt`, `updatedAt`

Rerun path: `GET /api/reports/:id/run` → load row → execute `sql` against readonly pool → return rows. **Zero Claude calls, zero token cost.**

Editing a saved report: opening its chat history and editing a chip prompts "Update saved report" vs "Save as new" (D12).

## No-SQL and no-IDs rules

### Never show SQL (D10)

Unchanged from prior direction. No preview, modal, toast, expandable section, dev-mode toggle, or error message containing SQL. pg errors are paraphrased by Claude (D5) before user-facing display.

Enforcement: code-review rule, plus a lint check that flags JSX rendering `result.sql` or `savedReport.sql` anywhere in `src/features/reports/components/`.

### No IDs in output (D11)

**Prompt layer:** system prompt includes: *"Never SELECT primary-key ID columns (leaid, opportunity_id, uuid, etc.) unless the user explicitly asked for them by name. Always prefer joining to the entity's name column. Example: if asked for 'districts', SELECT `districts.name`, not `districts.leaid`."*

**UI layer:** columns whose `ColumnMetadata.format === "id"` render hidden by default. A "Show technical columns" toggle in the results header reveals them. This is the backstop — even if Claude slips up, the rep doesn't see `3100009` in a result cell.
