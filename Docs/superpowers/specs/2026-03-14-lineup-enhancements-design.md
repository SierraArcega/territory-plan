# The Lineup Enhancements ‚Äî Design Spec

**Date:** 2026-03-14
**Branch:** `aston-lineup-tab` (or new branch from main post-merge)
**Status:** Design approved, pending implementation plan

---

## Overview

Two enhancements to The Lineup and activity editing experience:

1. **Edit gap fix** ‚Äî plan and state association is currently missing from the activity edit flow
2. **AI-powered goal suggestions** ‚Äî a smart recommendations panel in The Lineup that surfaces prioritised actions based on the user's stated goals and current progress

---

## Part 1 ‚Äî Edit Gap Fix

### Problem

When opening an existing activity to edit it, `ActivityFormModal` wraps both the Plans and States pickers in `{!isEditing && ...}` guards (lines 289 and 331), actively suppressing them in edit mode. It also resets `selectedPlanIds` and `selectedStateFips` to empty on mount. Users cannot see or modify plan/state associations after creation.

### Fix

1. **Remove the `!isEditing` guards** so plan and state fields render in edit mode
2. **Pre-populate from a full activity fetch** ‚Äî `ActivityListItem` only carries `planCount`, not IDs or fips. The edit modal calls `useActivity(initialData.id)` on open, guarded with `enabled: !!isEditing`, to load the full `Activity` type. `useActivity` is already available in `src/features/activities/lib/queries.ts` and requires no new import.
3. **Initialise selections** from the fetched data:
   - `selectedPlanIds` ‚Üê `activity.plans.map(p => p.planId)`
   - `selectedStateFips` ‚Üê `activity.states.filter(s => s.isExplicit).map(s => s.fips)`
   - Notes field ‚Üê `activity.notes` (populated once fetch resolves; shown as empty string until then ‚Äî no skeleton needed since empty is an acceptable interim state)
4. **`handleSubmit` in edit mode** is updated to pass `planIds: selectedPlanIds` and `stateFips: selectedStateFips` to `updateActivity.mutateAsync`

### Loading state (edit open)

While `useActivity(id)` loads, plan and state fields render in a disabled/skeleton state. Type, title, dates, status, and assignee (available immediately from `ActivityListItem`) render normally. Save button is disabled until fetch resolves.

### Save behaviour

`useUpdateActivity()` in `src/features/activities/lib/queries.ts` is extended to accept **optional** `planIds?: string[]` and `stateFips?: string[]`. Existing callers that omit these fields continue to work unchanged. The PATCH handler at `/api/activities/[id]` is extended to sync links only when these fields are present in the request body: remove deselected, add newly selected.

## Part 2 ‚Äî Rules-Based Goal Suggestions

### Solution

A persistent suggestions panel in The Lineup that uses a rules engine to surface prioritised recommended actions based on the user's `UserGoal` data, recent activity history, and active territory plans. **Shown only when `selectedDate === today`.** No AI API calls required ‚Äî suggestions are computed server-side at request time.

### Rules Engine

Up to 3 suggestions generated by evaluating rules in priority order, deduplicating by district. District ordering within each rule: contract value DESC, then `lastContactDays` DESC.

| Priority | Rule | Trigger | Suggested activity |
|---|---|---|---|
| 1 | **RENEWAL_BEHIND** | `renewalActual < renewalTarget √ó 0.9` | `call` ‚Äî "Renewal Check-in ‚Äî [District]" |
| 2 | **PIPELINE_BEHIND** | `pipelineActual < pipelineTarget √ó 0.9` | `call` ‚Äî "Pipeline Follow-up ‚Äî [District]" |
| 3 | **LONG_DORMANT** | No activity for this district in 45+ days, regardless of goals | `call` ‚Äî "Check In ‚Äî [District]" |
| 4 | **DEFAULT** | No goals set, or rules 1‚Äì3 produce fewer than 3 cards | Top high-value uncontacted districts |

Reasoning text is generated from templates referencing actual dollar gaps and days since last contact (e.g. "Your renewal goal is $12k behind ‚Äî [District] ($45k) hasn't been contacted in 38 days.").

---

### Layout

- A collapsed **banner** sits above the timeline when viewing today (hidden on all other dates)
- Clicking banner opens a **floating overlay** (does not push the timeline down)
- Overlay: goal context bar at top + suggestion cards below
- Closing returns to collapsed banner

**Busy day:** Busy = 4+ activities in the user's **own today activities** (filtered by `assignedToUserId = currentUser`, date = today, regardless of any person-selector or plan filter applied in the Lineup UI). This count is fetched independently or derived from the existing `useActivities` call scoped to today + current user. When busy, banner text: "Looks like your day is pretty booked ‚Äî click here if you need more ideas."

---

### Goal Context Bar

**If no `UserGoal` record exists for the current fiscal year**, replace the bar with: "Set your goals on the Home page to get personalised recommendations." Cards still show.

**Source fields:**

`UserGoal` actuals (`takeActual`, `pipelineActual`, etc.) are virtual fields returned by the API layer ‚Äî they are computed from activity and plan data at query time and are not persisted DB columns. The suggestions API computes these the same way the existing goals API does.

| Display | Target field | Actual / progress |
|---|---|---|
| Earnings | `earningsTarget` | `takeActual` (computed: sum of take amounts from completed activities in fiscal year) |
| Pipeline | `renewalTarget + winbackTarget + expansionTarget + newBusinessTarget` | `pipelineActual` (computed: sum of opportunity values in active pipeline) |
| Renewals | `renewalTarget` (dollar figure) | sum of completed renewal-type activity values in current fiscal year |

All values displayed as currency. No counts.

**Color logic:**
- < 90%: Coral gap label + bar
- 90‚Äì99%: Amber
- ‚â• 100%: Green + surplus shown

### Suggestion Cards

Up to 3 cards. If the rules engine produces fewer than 3, show however many are returned. If zero, show a single message: "Nothing urgent right now ‚Äî check back tomorrow."

Each card:

| Element | Detail |
|---|---|
| Activity type + icon | e.g. "üìû Renewal Call" |
| District + Plan names | from suggestion payload |
| Metrics row | 3 chips: contract/opportunity value, days since last contact, `renewalWeeks` (estimated from the date of the most recent renewal-type activity for that district plus a 12-month cycle ‚Äî null if no prior renewal activity exists) |
| Reasoning | 1‚Äì2 sentences from a template referencing goal gap and dollar amounts (e.g. "Your renewal goal is $12k behind ‚Äî this district ($45k) hasn't been contacted in 38 days.") |
| Tags | Goal category pills + risk status pills |
| Schedule button | Opens `ActivityFormModal` pre-filled with `activityType`, `title`, `planId`. A new optional `onSuccess: (activityId: string) => void` prop is added to `ActivityFormModal` ‚Äî it is called with the new activity's ID after `createActivity.mutateAsync` resolves. `SuggestionCard` passes this callback and uses the returned `activityId` to call `useLinkActivityDistricts().mutate({ activityId, leaids: [districtLeaid] })`. If this post-save link call fails: show a toast "Activity saved ‚Äî couldn't link district. Add it manually." The activity is kept. |

### Feedback Button

At the bottom of the overlay, below the suggestion cards:

> "Want smarter, AI-powered recommendations? [Let us know ‚Üí]"

Clicking the link:
1. POSTs to `POST /api/lineup/suggestions/feedback` (records `userId + createdAt` in `SuggestionFeedback` table)
2. Button state changes to "Thanks! We're tracking interest." (disabled, no reload)

This is a lightweight demand signal ‚Äî no personal data collected beyond the user ID and timestamp.

---

### Goal States

**Behind (< 90%):** Gap-focused, urgent, at-risk accounts prioritised.
**Nearing (90‚Äì99%):** Motivating push to cross the line.
**Exceeding (‚â• 100%):** Upside opportunities ‚Äî additional earnings potential, no gap language.

---

### Data Fetching

New hook: `useLineupSuggestions(date: string)`:
- Returns `null` immediately if `date !== today`
- Otherwise calls `GET /api/lineup/suggestions`
- Returns `{ suggestions, isLoading, error, refetch }`

---

### Loading & Error States

- Fires on mount when viewing today
- Banner shows shimmer while `isLoading`
- No server-side caching needed ‚Äî the rules engine is a pure DB query + computation
- **Error:** Banner shows "Couldn't load recommendations ‚Äî try again later" + retry button (calls `refetch()`)

---

### Suggestions API

**Endpoint:** `GET /api/lineup/suggestions`

**Server-side computation:**
1. Load `UserGoal` for current fiscal year + compute actuals (same logic as existing goals API)
2. Load last 30 days of user's activities (type, district leaid, outcome, date)
3. Load active territory plans + districts (names, leaids, contract values)
4. Run rules engine (see Rules Engine section above)
5. Return suggestion array

**Feedback Endpoint:** `POST /api/lineup/suggestions/feedback` ‚Äî records a `SuggestionFeedback` row for `userId`.

**Output shape:**

```ts
{
  activityType: string
  title: string
  districtLeaid: string | null
  districtName: string | null
  planId: string | null
  planName: string | null
  contractValue: number | null
  lastContactDays: number | null
  renewalWeeks: number | null          // weeks since most recent renewal activity + 52-week cycle; null if no prior renewal
  opportunityType: "renewal" | "expansion" | "winback" | "new_business"
  reasoning: string                    // template-generated, not AI
  goalTags: string[]
  riskTags: string[]
}
```

## New DB Model

```prisma
// Tracks user interest in AI-powered suggestions ‚Äî used as a demand signal.
model SuggestionFeedback {
  id        String      @id @default(uuid())
  userId    String      @map("user_id") @db.Uuid
  user      UserProfile @relation(fields: [userId], references: [id], onDelete: Cascade)
  createdAt DateTime    @default(now()) @map("created_at")

  @@index([userId])
  @@map("suggestion_feedback")
}
```

`UserProfile` gains: `suggestionFeedback SuggestionFeedback[]`
