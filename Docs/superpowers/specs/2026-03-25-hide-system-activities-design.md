# Hide System Activities from Frontend Views

**Date:** 2026-03-25
**Status:** Approved

## Problem

Bulk Contact Enrichment creates `contact_enrichment` activities with `source: "system"`. These show up in all activity lists alongside user-created activities, adding noise to the Activities tab. System-generated activities are internal operational records — they're useful for auditing via direct database queries but shouldn't be surfaced to end users.

## Decision

Filter system activities at the API level. The `GET /api/activities` endpoint will exclude activities with `source = 'system'` by default. This is a single change point that covers all frontend consumers automatically.

## Changes

### 1. API Filter — `src/app/api/activities/route.ts`

Add `source: { not: 'system' }` to the Prisma `where` clause in the GET handler. All existing and future system-sourced activities are immediately hidden from every frontend view.

### 2. Remove System Types from UI — `src/features/activities/types.ts`

Remove the `system` category from `ACTIVITY_CATEGORIES` and remove `contact_enrichment` from:
- `ACTIVITY_TYPE_LABELS`
- `ACTIVITY_TYPE_ICONS`
- Any other display mappings

This prevents system types from appearing in creation forms, filter chips, or type dropdowns. The bulk-enrich API route continues writing `type: "contact_enrichment"` directly — it doesn't rely on these UI constants.

### 3. Verify Filter Chips — `src/features/activities/components/ActivityFilterChips.tsx`

Confirm "system" is not listed as a source filter option. Currently it only shows: All, Email, Calendar, Slack, Manual. No change expected, but verify.

### 4. README — Activity Source Types Documentation

Add a section to the project README documenting activity source types and their visibility:

| Source | Origin | Visible in UI |
|--------|--------|---------------|
| `manual` | Created by users via the Activity Form | Yes |
| `calendar_sync` | Imported from Google Calendar | Yes |
| `gmail_sync` | Imported from Gmail | Yes |
| `slack_sync` | Imported from Slack | Yes |
| `system` | Generated by platform operations (e.g., bulk enrichment) | No — database-only |

## What Stays the Same

- **Bulk-enrich API** (`POST /api/territory-plans/[id]/contacts/bulk-enrich`) — continues creating `contact_enrichment` activities with `source: "system"`
- **Enrichment progress polling** (`GET /api/territory-plans/[id]/contacts/enrich-progress`) — unchanged, references activity by ID directly
- **Single activity fetch** (`GET /api/activities/[id]`) — no filter added; internal routes that reference a specific activity by ID still work
- **Database records** — all system activities preserved, queryable via SQL/Prisma

## Future Extensibility

- Any new system-generated activity type using `source: "system"` is automatically hidden
- If admin views are ever needed, add an `?includeSystem=true` query param to the list endpoint
